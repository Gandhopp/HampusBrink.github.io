Ternion

Ternion is the name of the game I made for my graduation project during my time studying at Forsbergs.
Together with two of my classmates Luca Belluso and Raphael Mccabe we came up with the id√© of making a multiplayer first person shooter in Unreal Engine 5.
The video on the home page is also from this game.

My main goal with this project was that I really wanted to learn networking code since I've always preferred multiplayer games over single player games.
So naturally I became the guy responsible for every single network related code for the project.
Since we were only three people taking on the task of creating a 3v3 competitive shooter I also had to take part in creating other vital parts of the game like an easy to use Gun creation system,UI, VFX, SFX and some other stuff but my primary focus was networking.

Network Overview

The core of our multiplayer system is built around Epic Games networking tool: Epic Online Services (EOS) which is an SDK for Unreal Engine that has built in features like Session creation, Session finding and joining and most importantly allows us to connect to our game through Epic Games accounts.
I would split up everything network related that I made for the game into two parts, the first half is the external part, setting up EOS to work with Unreal Engine and actually connect to clients to the same server.
The second half, replication, is a more traditional style of programming such as c++ code and blueprints which can all be done within the game engine without anything external.
Due to very poor documentation of the SDK I turned to tutorials explaining EOS to someone like me who have no prior experience with the tool.
Unfortunately these tutorials ended rather abruptly without showing how to actually get two players on each screen (kinda important if you want multiplayer).
It took me about a week of searching and annoying my teachers and their previous colleagues before I finally could get it to work.
Unfortunately for me the replication part was not much easier.
Understanding what events needed to be called from the server and which clients owned which objects and what objects existed on what client was very confusing at first.
For example I didn't know that the server knew about every playercontroller but clients only knew their own.
This caused the server to duplicate its UI when accessing it from the player controller as it received the call multiple times and changed the UI that it was not supposed to have which was just a mess.
Issues like these appeared all the time, Walls spawning for server but not for clients, damage sometimes applied twice for the server but not on the clients, victory ui only showing for exactly 2 people but never more never less and many more.
While yes, fixing these issues were incredibly frustrating due to how different you have to approach problems compared to your regular programming issues.
Learning what information the server should handle, Understanding why the RPC (remote procedure call) call doesn't reach the client due to the object only existing on the server and finally fixing it was not only super satisfying but also saved the project all together.
Just thinking about it makes me want to keep on working on this project.

Gun System

Another thing I was responsible for making was a system that allowed us to create easily modifiable guns for every character.
One thing in particular that we wanted to have in the game was a recoil pattern that wasn't going to be random but the same every time.
I love when fps games have a really difficult recoil pattern just because it's a unique but effective way to make shooting more difficult than just requiring massive amounts of aim practice.
If you've ever played the game Rust then you know what I'm talking about.
Rust has a weapon that has such a difficult spray pattern that you need thousands of hours to perfect it.
Unfortunately they reworked this gun to make it easier for newer players and a big inspiration for me to make the guns in our game so difficult to control was because of my desire to recreate the feeling I got from mastering that gun.
Our gun system comes with a tedious but necessary array that you put in x and y values for where you want each individual bullet to land.
While most recoil pattern systems use a curve to make the bullets go in a general pattern I really wanted to have the freedom of making sure I have full control over every individual bullet and to achieve this I thought that an array would be the optimal answer.
And we only have 3 weapons, one being a sniper so technically only 2 that we needed to make an array for so not that much time was lost.

UI

I made the groundwork for the HUD of the game as well.
I'm no designer so Raphael had to make it pretty but the functionality is my doing.
I'm especially proud of the making of the kill feed and scoreboard as they both required quite a bit of refactoring of my code to credit each player correctly.
I also had to make sure that you wouldn't get a kill from suicide and that killing a team mate wouldn't get you kill credit either.
Updating every player when someone died was also not as easy as you might think because of the fact that widgets (UI objects in unreal) do not receive RPC calls as they only exist locally.
I also designed these two so my artistic skills are not completely nonexistent.
